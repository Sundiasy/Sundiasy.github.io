您提供的这个信息非常关键！”安装在了user目录下 ~/ollama/bin“ 这句话完全改变了解决问题的方向，也完美解释了为什么之前的命令会失败。

这说明：

  1. 这是一个纯粹的用户级安装，不涉及系统目录，因此它必须作为 用户服务 (User Service) 来运行。
  2. 之前 systemctl --user edit ollama.service 失败的原因，就是因为安装程序没有自动创建这个用户服务文件。
  3. 我们不应该使用 sudo systemctl，因为它用于管理系统服务，而去操作一个位于用户家目录下的程序，可能会引发权限问题，并且不符合规范。

正确的解决方案是：我们手动为当前用户创建一个 systemd 用户服务文件。

这很简单，请一步步跟我来操作。

详细步骤：手动创建并配置 Ollama 用户服务

步骤 1: 创建 systemd 用户服务目录 (如果不存在)

首先，确保存放用户服务文件的目录存在。这个命令很安全，如果目录已存在，它不会做任何事。

步骤 2: 创建并编辑 ollama.service 文件

现在，我们使用文本编辑器 (这里用 nano，因为它简单易用) 来创建这个服务文件。

步骤 3: 粘贴服务配置内容

将下面的所有内容完整地复制并粘贴到 nano 编辑器中。

重要提示：我已经根据您提供的路径 ~/ollama/bin (即 /home/sr/ollama/bin) 和您的GPU配置需求，为您定制好了这份配置。

配置解释：

  • ExecStart=/home/sr/ollama/bin/ollama serve: 这是核心！它告诉 systemd 你的 ollama 程序的确切位置。我已将 ~ 展开为 /home/sr，因为 systemd 需要绝对路径。
  • Environment="CUDA_VISIBLE_DEVICES=0,1": 在这里修改为您想使用的GPU！
    • 用一个GPU (第0号): "CUDA_VISIBLE_DEVICES=0"
    • 用两个GPU (第2和3号): "CUDA_VISIBLE_DEVICES=2,3"
  • Restart=always: 确保服务如果意外崩溃，会自动重启。

编辑完成后，按 Ctrl + X，然后按 Y 保存，最后按 Enter 确认文件名并退出。

步骤 4: 重新加载、启用并启动服务

现在文件已经创建好了，我们需要让 systemd 读取它，并启动服务。所有命令都带 --user，不需要 sudo。

步骤 6: (重要) 启用 Linger，让服务在您登出后继续运行

默认情况下，当您退出 SSH 连接时，您的所有用户服务都会被终止。为了让 Ollama 服务在您登出后依然保持运行，需要为您的用户启用 “Linger”。

这个命令只需要执行一次，之后就永久生效了。$(whoami) 会自动替换成您当前的用户名 (sr)。

总结

现在，您的 Ollama 已经作为一个健壮的、可配置的、持久化的用户服务在后台运行了。您随时登录 ssh node04，都可以直接使用 ollama run、ollama list 等命令，并且它会严格按照您在 ollama.service 文件中配置的 GPU 来运行。

如果您未来想更改 GPU 配置，只需：

  1. systemctl --user edit ollama.service (现在这个命令可以用了！)
  2. 修改 Environment 行。
  3. systemctl --user daemon-reload
  4. systemctl --user restart ollama

好的，你观察到的现象非常关键！

systemctl cat ollama.service 的结果只显示了原始的服务文件内容，而没有显示你在 systemctl edit 中添加的覆盖配置。

这几乎可以 100% 确定问题出在编辑器保存环节。虽然你用了 sudo，命令本身有权限，但你进入编辑器后，所做的修改最终没有被成功写入到磁盘上的覆盖文件 (override.conf) 中。

为什么会这样？

最常见的原因是：

  1. 你使用的默认编辑器不是 nano，而是 vi 或 vim。vi/vim 的保存退出命令不是 Ctrl+X，而是 按下Esc键 -> 输入 :wq -> 回车。如果你在 vim 里按 Ctrl+X，它什么也不会做。
  2. 即使是 nano，也可能因为某些终端问题或误操作导致保存失败。

终极解决方案：绕过编辑器，直接创建配置文件

既然交互式的编辑器出了问题，我们就用一个更直接、更不容易出错的方法：通过命令直接生成配置文件。这个方法可以绕过所有编辑器的不确定性。

请按顺序执行以下命令，这将保证配置被正确创建。

第 1 步：创建覆盖配置目录

systemd 的覆盖配置文件放在一个与服务同名的 .d 结尾的目录里。我们先确保这个目录存在。

这条命令做了什么？

  • sudo tee ... : 以 root 权限运行 tee 命令，它会从输入中读取内容并写入文件。这是在需要 root 权限时向文件写入内容的最可靠方法。
  • <<EOF ... EOF : 这是一个 “here document”，它将 EOF 之间的所有文本作为输入传给前面的命令。
  • > /dev/null : tee 默认也会把内容输出到屏幕，这个部分是把它屏幕输出丢掉，让界面更干净。

第 5 步：最终验证

现在，当 Ollama 服务启动后，它已经被限制只能使用 GPU 1 了。

  1. 运行一个模型: ollama run llama3
  2. 在另一个终端查看: nvidia-smi

你应该会看到 ollama 的进程只出现在 GPU 1 下方。

这个“直接写入文件”的方法比“编辑文件”更可靠，尤其是在不确定默认编辑器行为的情况下。请试试这个流程，它应该能解决你的问题。

mkdir -p ~/.config/systemd/user/ollama.service.d/

nano ~/.config/systemd/user/ollama.service.d/override.conf

systemctl –user daemon-reload

systemctl –user restart ollama.service
